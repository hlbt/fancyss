# SSEP：订阅会话加密协议标准草案 v1

本文用于描述一种“订阅内容按请求临时加密、客户端按本次会话临时解密”的跨客户端协议草案。

当前状态：草案，不是最终版。

推荐工作名：

- 中文名：订阅会话加密协议
- 英文缩写：`SSEP` (`Subscription Session Encryption Protocol`)

命名说明：

- 名称不绑定 `fancyss`
- 名称不绑定 `User-Agent`
- 名称不绑定某一种加密算法

这样后续即使从 `UA` 迁移到独立 HTTP Header、从 `AES-256-GCM` 增加到 `ChaCha20-Poly1305`，标准名也不需要变化。

---

## 一、背景与设计动机

当前常见订阅链接存在一个明显问题：

- 订阅内容通常是明文文本、Base64 文本、JSON、YAML
- 拿到订阅链接后，任何人都可以直接获取订阅内容
- 再进一步解析出节点地址、域名、中转机 IP 等敏感基础设施信息

这使得“拿到订阅链接即可低成本批量收集节点地址”成为现实。

SSEP 的核心目标不是“让节点地址对客户端永远不可见”，而是：

1. 让订阅链接本身不再直接暴露明文节点内容
2. 让服务端可以只对“支持 SSEP 的客户端会话”返回可解密的密文订阅
3. 降低订阅链接泄露、网页打开、普通脚本抓取、被动解析等带来的明文暴露风险
4. 为后续“前端隐藏地址、存储层加密、日志脱敏”提供协议基础

---

## 二、威胁模型与明确边界

### 2.1 本草案试图降低的风险

- 订阅链接被转发、截图、粘贴后，第三方直接请求得到明文订阅
- 用浏览器、`curl`、`wget`、通用脚本直接获取并解析订阅
- 普通泄露场景中，订阅链接一旦外流就立即暴露全量节点地址
- 不支持 SSEP 的通用客户端把订阅当作明文内容直接解析

### 2.2 本草案不能根治的风险

以下风险无法仅靠“客户端发临时 key，服务端按此加密”彻底解决：

- 用户完全控制自己的客户端进程
- 用户修改开源客户端，在生成会话密钥时直接打印出来
- 用户抓包看到自己发出的会话密钥
- 用户在客户端解密后，从内存、日志、调试接口、运行配置中提取真实地址
- 用户在客户端 UI 中逐个查看节点详情
- 已 Root / 越狱 / 有本机调试权的终端直接读取客户端运行态数据

因此，SSEP 的准确定位应是：

- 它能显著提高“低成本批量抓取明文订阅”的门槛
- 它不能保证“恶意付费用户在完全控制终端的前提下永远拿不到节点地址”

这点必须在所有实现和宣传中写清楚，避免错误预期。

---

## 三、总体设计原则

1. 协议标准与具体客户端解耦。
2. 协议标准与具体承载方式解耦。
3. 协议标准与具体实现库解耦。
4. 明文订阅与加密订阅必须可共存。
5. 第一版必须预留版本升级空间。
6. 对已有解析器影响尽量小。
7. 优先保护“订阅传输过程中的明文暴露”，再逐步扩展到“展示层和存储层保护”。

## 四、术语

- `Client`
  发起订阅下载并负责解密的客户端，例如 `fancyss`、`merlinclash`、`passwall`、PC 客户端、手机客户端。

- `Provider`
  提供订阅服务的服务端。

- `Session Secret`
  客户端为本次订阅请求随机生成的临时高熵随机值，仅用于当前请求。

- `Carrier`
  客户端把协议信息带给服务端的承载方式，例如 `HTTP Header` 或 `User-Agent`。

- `Envelope`
  服务端返回的加密订阅外层封装对象。

- `Plain Payload`
  被加密之前的原始订阅内容字节流。

- `Protected Subscription`
  指服务端返回的 SSEP 加密封装订阅。

- `Legacy Subscription`
  指传统明文 / Base64 / JSON / YAML 订阅。

## 五、协议定位

SSEP 只定义：

- 客户端如何声明“我支持会话加密订阅”
- 客户端如何把本次会话所需参数发送给服务端
- 服务端如何封装加密响应
- 客户端如何判断、解密并恢复出原始订阅内容

SSEP 不直接规定：

- 节点如何在 UI 展示
- 节点如何在本地数据库中长期保存
- 节点运行配置如何避免被本机读取
- 服务端订阅鉴权模型

这些内容属于上层实现策略，但可在后续文档中给出推荐做法。

## 六、版本与兼容策略

### 6.1 版本字段

第一版协议版本号固定为：

- `v=1`

### 6.2 向后兼容规则

- `Client` 必须忽略自己不认识的扩展字段
- `Provider` 必须允许根据 `v` 选择不同解析路径
- 后续新增字段应采用“可选字段 + 默认行为”的方式演进
- 第一版不允许在同一版本号下改变已定义字段语义

### 6.3 注册表预留

建议后续为以下字段建立小型注册表：

- `alg`：加密算法
- `zip`：压缩方式
- `fmt`：明文订阅格式
- `carrier`：协议信息承载方式

## 七、承载方式

### 7.1 总原则

标准层面不应把“会话密钥放在 UA 中”写死为唯一方式。

原因：

- `User-Agent` 很容易被 Web 服务器、中间件、CDN、WAF、反向代理、日志系统完整记录
- `User-Agent` 语义上本就不是密钥承载字段
- 某些客户端或系统对 `UA` 长度、字符集、格式兼容性较差

因此，第一版协议分为：

- 首选承载：独立 HTTP Header
- 兼容承载：`User-Agent` 附加 profile

### 7.2 首选承载：独立 Header

推荐头：

- `X-SSEP`

示例：

```http
X-SSEP: v=1;k=2vQm2P0D4lM6m9Vt4r9P6Gf5X0QkQv8q6eQw8M2m1XA;req=9uFhXhJ0-9W2Kx3Q;alg=A256GCM;client=fancyss/3.3.9
```

字段说明：

- `v`
  协议版本，当前固定为 `1`
- `k`
  本次订阅的 `Session Secret`，使用 `base64url` 编码，无 `=`
- `req`
  本次请求随机 ID，使用 `base64url` 编码，无 `=`
- `alg`
  客户端请求的首选算法，第一版固定 `A256GCM`
- `client`
  客户端标识，便于服务端兼容处理，不参与密钥计算

### 7.3 兼容承载：UA Profile

某些客户端初期只方便改 `UA`，因此允许定义 `UA` 兼容 profile。

推荐格式：

```text
AsusWRT|koolcenter|RT-AX86U|3004_xxx|fancyss|hnd|full|3.3.9 SSEP/1 k=2vQm2P0D4lM6m9Vt4r9P6Gf5X0QkQv8q6eQw8M2m1XA req=9uFhXhJ0-9W2Kx3Q alg=A256GCM
```

约束：

- `SSEP/1` 为固定前缀
- 参数采用 `key=value`
- 值仅允许使用 URL-safe Base64、字母、数字、`.`、`-`、`_`
- 不使用空格以外的复杂分隔符

注意：

- `UA` profile 仅作为兼容方案，不建议作为标准长期默认方案
- 服务端和中间层如记录完整 `UA`，等价于记录了本次会话密钥

### 7.4 日志要求

无论使用 Header 还是 UA：

- `Provider` 端访问日志不得原样记录 `k`
- `Client` 端日志不得输出 `k`
- 反向代理 / CDN / WAF / 应用日志需要对 `k` 做脱敏或关闭记录

## 八、随机数、密钥与算法选择

### 8.1 会话密钥类型

第一版采用：

- 对称加密
- 每次请求一个新的随机 `Session Secret`

原因：

- 订阅下载本身通常基于 `HTTPS`
- 服务端已经能直接看到客户端发来的会话参数
- 在这种前提下，再引入非对称加密并不能从根本上解决“客户端最终可解密”的问题
- 对称方案更简单、更适合多平台落地

### 8.2 会话密钥长度

客户端必须生成：

- `32` 字节随机值作为 `Session Secret`

再做 `base64url` 编码放入请求。

不要：

- 直接使用短字符串
- 使用时间戳、UUID、设备序列号等低熵数据
- 让用户手输密码

### 8.3 第一版强制算法

第一版建议的 `MUST` 算法：

- `A256GCM`
  即 `AES-256-GCM`

原因：

- 算法成熟
- 通用库支持广
- 同时具备机密性和完整性校验

### 8.4 后续可选算法

后续版本可扩展：

- `CHACHA20-POLY1305`

但不建议在第一版就把多算法协商做复杂。

### 8.5 盐、IV 与 KDF

不要把客户端传来的 `Session Secret` 直接当作 AES key 使用。

第一版要求：

1. `Provider` 为每个响应生成 `salt`
   - `16` 字节随机值
2. `Provider` 为每个响应生成 `iv`
   - `12` 字节随机值，供 `AES-GCM` 使用
3. 使用 `HKDF-SHA256` 从 `Session Secret` 派生真正的内容加密密钥

推荐派生方式：

- `IKM = SessionSecret(32 bytes)`
- `salt = response.salt`
- `PRK = HKDF-Extract(salt, IKM)`
- `enc_key = HKDF-Expand(PRK, "SSEP-1 A256GCM key", 32)`

说明：

- 这里的 `salt` 已经承担了第一版所需的“加盐”作用
- 不需要再额外引入兼容性差的 `openssl enc` 口令式 `salt+password` 模式
- 协议层应该使用“随机 key + HKDF”的现代做法，而不是“用户口令 + 老式 PBKDF”做法

## 九、明文订阅的表示方式

### 9.1 原则

应先确定“订阅原始明文字节流”，再对该字节流进行可选压缩和加密。

推荐顺序：

```text
原始订阅内容 -> 可选压缩 -> AEAD 加密 -> base64url 封装为 JSON 字段
```

不推荐：

```text
先把业务明文做一次无意义的 base64 -> 再加密 -> 再外层 base64
```

### 9.2 对问题 1 的明确回答

对于“加密后再 Base64，还是 Base64 后再加密”，第一版建议是：

- 业务明文不需要为了加密而先做一次 Base64
- 应直接对原始订阅明文字节流加密
- 因为密文是二进制，所以外层封装时再使用 `base64url`

也就是：

```text
原始明文 -> 加密 -> 外层 base64url
```

### 9.3 明文格式字段

Envelope 中需要显式带上原始明文格式：

- `fmt=ss-b64-lines`
- `fmt=uri-lines`
- `fmt=clash-yaml`
- `fmt=singbox-json`
- `fmt=raw`

说明：

- `fmt` 表示“解密后应该按什么方式继续走原有解析器”
- 客户端解密成功后，不必重新猜格式

### 9.4 压缩字段

大订阅可选压缩：

- `zip=none`
- `zip=gzip`

第一版建议：

- 小订阅默认 `zip=none`
- 对较大的 YAML / JSON 订阅允许 `gzip`

## 十、响应 Envelope 定义

### 10.1 Content-Type

服务端返回加密订阅时，建议使用：

```http
Content-Type: application/ssep+json
Cache-Control: no-store, private
Pragma: no-cache
```

### 10.2 JSON 结构

第一版建议 Envelope 结构如下：

```json
{
  "v": 1,
  "alg": "A256GCM",
  "fmt": "uri-lines",
  "zip": "none",
  "salt": "a5K9Y1m2I6qV1YH4mM2D5A",
  "iv": "0wQjL6qP8zT1mV2N",
  "req": "9uFhXhJ0-9W2Kx3Q",
  "sid": "n4rM8QvD3pL1cT7Z",
  "ts": 1775140800,
  "ttl": 30,
  "ct": "....",
  "tag": "...."
}
```

字段说明：

- `v`
  协议版本
- `alg`
  加密算法
- `fmt`
  解密后订阅原始格式
- `zip`
  压缩方式
- `salt`
  HKDF 所需随机盐，`base64url`
- `iv`
  `AES-GCM` 随机 IV，`base64url`
- `req`
  回显客户端请求中的 `req`
- `sid`
  服务端生成的响应会话 ID，便于日志排障
- `ts`
  服务端生成此响应的 Unix 时间戳
- `ttl`
  此响应建议有效期，单位秒
- `ct`
  密文字节，`base64url`
- `tag`
  GCM 认证标签，`base64url`

### 10.3 AAD

第一版建议把以下字段拼成规范化字符串作为 `AAD`：

```text
v=1&alg=A256GCM&fmt=uri-lines&zip=none&req=9uFhXhJ0-9W2Kx3Q&sid=n4rM8QvD3pL1cT7Z&ts=1775140800&ttl=30
```

要求：

- 字段顺序固定
- 使用 ASCII
- 不得省略已定义字段

这样客户端可以同时校验：

- 密文是否被篡改
- `req` 是否与当前请求匹配
- 时间窗是否合理

## 十一、协议流程

### 11.1 标准流程

1. `Client` 生成 `32` 字节随机 `Session Secret`
2. `Client` 生成本次请求随机 `req`
3. `Client` 通过 `X-SSEP` 或 `UA Profile` 发送：
   - `v`
   - `k`
   - `req`
   - `alg`
   - `client`
4. `Provider` 检测当前请求是否带有有效的 SSEP 参数
5. 若支持且允许当前用户使用 SSEP：
   - 生成原始订阅明文
   - 选择 `fmt`
   - 选择是否压缩
   - 生成 `salt`
   - 生成 `iv`
   - 用 `HKDF-SHA256` 派生 `enc_key`
   - 用 `AES-256-GCM` 加密
   - 组装 Envelope JSON 返回
6. `Client` 收到响应后，先判断是否为 `application/ssep+json`
7. 若是：
   - 解析 Envelope
   - 验证 `v`
   - 验证 `req`
   - 验证 `ts + ttl`
   - 使用 `salt + Session Secret` 派生 `enc_key`
   - 使用 `AAD` 验证并解密
   - 若 `zip=gzip` 则先解压
   - 把得到的明文订阅内容交给现有解析器
8. 解析完成后，`Client` 销毁本次 `Session Secret`
9. 若不是加密响应，则按 Legacy 订阅处理

## 十二、错误处理与回退策略

### 12.1 客户端模式建议

建议客户端提供三种模式：

- `plain-only`
  只接受传统明文订阅
- `auto`
  优先尝试 SSEP，失败时可回退到传统订阅
- `protected-only`
  必须成功走 SSEP，否则视为失败

### 12.2 回退规则建议

在 `auto` 模式下：

- 服务端完全不支持 SSEP
  可回退
- 服务端返回明文 Legacy 订阅
  可回退
- 服务端返回了 SSEP Envelope，但解密失败
  不建议静默回退

原因：

- “已进入受保护流程但解密失败”通常意味着实现不匹配、响应损坏、算法不兼容或中间层篡改
- 这类情况下静默回退到明文，容易掩盖真实问题

### 12.3 服务端错误码建议

第一版可选：

- `400`
  SSEP 参数格式非法
- `406`
  当前客户端请求的算法不支持
- `409`
  当前用户未启用受保护订阅
- `426`
  协议版本过低

但即使没有这些状态码，只要 Envelope 可识别、错误可诊断即可。

## 十三、安全要求

### 13.1 传输要求

- 必须运行在 `HTTPS`
- 不应在明文 HTTP 下使用 SSEP

### 13.2 密钥要求

- `Session Secret` 只允许保存在本次订阅进程内存中
- 不写持久化文件
- 不写 dbus
- 不写全局环境变量
- 不打印到日志

### 13.3 客户端临时文件要求

客户端实现时建议：

- 尽量避免把“已解密的完整订阅明文”长期缓存到磁盘
- 如必须落盘，应使用临时目录并在完成后清理
- 加密订阅与明文订阅缓存要分开标记

### 13.4 服务端日志要求

- 不记录 `k`
- 不记录完整 `req`
- 避免在错误日志中输出完整 Envelope 与请求参数

### 13.5 共享缓存要求

服务端如使用 CDN / 反代 / 网关：

- 必须避免不同用户之间共享受保护订阅缓存
- 必须正确处理 `Vary` 头
- 必须确认中间层不会按 `UA` 或 Header 记录完整敏感参数

## 十四、多平台实现建议

### 14.1 标准层不要绑定命令行工具

标准应定义：

- 密钥生成要求
- HKDF 派生要求
- AEAD 加密要求
- Envelope 格式

而不是定义：

- 必须用某个命令行参数
- 必须用某个 shell 管道

### 14.2 对“是否必须用 openssl”的建议

对用户提出的建议，第一版的结论是：

- 协议层不绑定 `openssl`
- 实现层可以使用 `OpenSSL`、`LibreSSL`、`BoringSSL`、系统加密 API、成熟的静态链接小库

原因：

- 跨平台标准不应依赖某个 CLI 的参数细节
- 很多嵌入式环境的 `openssl` CLI 对 `AEAD`、`HKDF`、参数格式支持并不完全一致
- 协议要能被路由器、Windows、macOS、Android、iOS 统一实现

结论：

- `OpenSSL` 可以作为若干客户端的实现选项
- 但它不应成为标准的一部分

## 十五、关于“客户端生成密钥后即销毁”的现实说明

“本次订阅的随机密钥不落盘、只存在于进程内存、订阅完成后即销毁”是正确且必要的工程要求。

但这并不等于：

- 恶意用户就无法得到该密钥

因为如果用户完全控制自己的客户端环境，他仍然可以：

- 修改客户端，在生成后立即打印密钥
- 用调试器查看内存
- 抓取自己设备发出的请求

因此，第一版标准必须把目标明确写成：

- 抬高大规模低成本批量抓取门槛
- 降低明文链接泄露的破坏面

而不是：

- 彻底阻止终端所有者获取自己终端已经解密过的数据

---

## 十六、与 UI 隐藏、存储加密的关系

SSEP 只解决“订阅下发过程”的保护问题。

若要进一步提高保护强度，还需要配套：

1. UI 不直接显示受保护节点的真实地址
2. 日志不打印受保护节点的真实地址
3. 导出 / 备份时对受保护节点做额外限制
4. 本地存储尽量不以明文保存敏感地址字段
5. 运行时只在真正需要生成配置时短暂解密

这些都不属于 SSEP 协议本体，但应作为客户端落地建议同步推进。

---

## 十七、对用户提出的三个问题的结论

### 17.1 订阅内容应该如何编码

结论：

- 业务明文直接加密
- 密文二进制再做外层 `base64url`

即：

```text
明文 -> 可选压缩 -> 加密 -> base64url
```

### 17.2 对称还是非对称

结论：

- 第一版优先采用对称方案

原因：

- 当前场景基于 `HTTPS`
- 服务端本来就能看到会话参数
- 非对称方案并不能自动解决“客户端终端可提取”的根本问题
- 对称方案更适合第一版快速跨平台落地

### 17.3 算法、key 长度、是否加盐

结论：

- 算法：`AES-256-GCM`
- 会话原始随机值长度：`32` 字节
- `salt`：需要，`16` 字节随机值
- `iv`：需要，`12` 字节随机值
- KDF：`HKDF-SHA256`

不要：

- 把短密码直接喂给 AES
- 使用口令式弱派生
- 复用固定 IV

## 十八、第一版推荐的最小互通子集

为了让不同客户端尽快跟进，建议先定义一个最小子集：

### Client 必须支持

- `v=1`
- `alg=A256GCM`
- `fmt=uri-lines`
- `zip=none`
- Header 承载
- UA 兼容 profile 解析

### Provider 必须支持

- 识别 `v=1`
- 接受 `A256GCM`
- 返回 `application/ssep+json`
- 正确回显 `req`
- 使用 `HKDF-SHA256 + AES-256-GCM`

等生态稳定后，再扩展：

- `clash-yaml`
- `singbox-json`
- `gzip`
- 新算法
- 设备绑定增强

---

## 十九、后续增强方向

第一版之后，可继续研究：

1. 设备绑定密钥
2. 安全硬件 / 安全存储
3. 服务端按设备发唯一节点集
4. 水印订阅 / 蜜罐节点
5. 节点地址分层下发
6. UI 与导出能力按受保护级别差异化处理

但这些都不应阻塞第一版的“传输层会话加密”落地。

---

## 二十、结论

SSEP v1 的本质是：

- 用一次性高熵会话密钥，把“可直接明文获取的订阅”改造成“仅支持 SSEP 的会话才能恢复的订阅”

它适合做：

- 订阅传输保护的第一阶段标准
- 多客户端共用的最小交集协议
- 后续展示层、存储层保护的基础能力

它不应被宣传为：

- 终端所有者绝对无法提取节点地址的最终方案

第一版如果按以上边界推进，是值得做的。
