# 节点身份机制 TODO

## 背景

当前 fancyss 在订阅前后，已经能够：

- 用 schema2 稳定 ID 保存“本地节点身份”
- 在订阅重写后尝试恢复：
  - 当前使用节点
  - 故障转移备用节点

但这套恢复仍然主要依赖当前节点的几个显式字段：

- `name`
- `type`
- `server`
- `port`

这在很多机场“节点名字稳定、但连接参数会变化”的场景下还不够强。

另外，节点分流上线后，这个问题会更关键：

- 分流规则卡片里绑定的是某个出口节点
- 订阅后节点顺序可能变化
- 节点名字、协议、端口、域名也可能变化
- 如果不能识别“这是同一个节点的更新”，原有映射关系就会断

## 典型问题

### 1. 顺序变化

用户当前使用的是：

- 主节点：香港 09
- 故障转移节点：日本 03

订阅更新后：

- 节点顺序整体重排

如果只看旧顺序号，那么：

- 当前节点引用会漂移
- 备用节点引用也会漂移

### 2. 同名节点参数变化

机场可能维持同一个展示名：

- `香港 09`

但每次订阅下发的：

- `server`
- `port`
- `relay server`
- `tuic_json`
- `xray_json`

会发生变化。

这时它对用户来说更像“同一个节点更新了参数”，而不是“新节点”。

### 3. 名字变化但连接参数不变

机场可能把：

- `香港 09`

改成：

- `香港 09 [Premium]`

或者：

- `HK 09`

如果连接参数基本不变，那么更合理的判断仍然应该偏向：

- 这是原节点的重命名

### 4. 协议切换

有些机场可能把同一落地从：

- `vmess`

换成：

- `vless`

甚至：

- `tuic`

这时是否视作同一节点，需要一套可解释的规则。

## 目标

节点身份机制未来至少要支持：

1. 订阅前后识别“这是同一个节点更新了参数”
2. 在订阅重写后恢复：
   - 当前使用节点
   - 故障转移备用节点
   - 节点分流卡片绑定的出口节点
3. 在节点被删除时，为主节点/备用节点/分流出口节点选择合理替代
4. 尽量减少“只因顺序变化或名称微调就丢失映射”的情况

## 建议的身份层级

不要只用一个规则，而要分层匹配。

### Level 1: 强身份

优先命中条件：

- 协议相同
- 原始名称相同
- 关键连接参数高度相似

适用：

- 大多数“同名节点更新参数”场景

### Level 2: 名称优先身份

条件：

- 原始名称相同
- 协议允许切换
- 关键参数发生变化

适用：

- 机场保留同名，但切换协议或端口

### Level 3: 参数优先身份

条件：

- 名称发生变化
- 但关键连接参数主体未变

适用：

- 节点重命名
- 机场增加 `[Premium]`、倍率标签、地区前缀

### Level 4: 近邻替代

当原节点完全消失时：

- 同地区
- 同协议优先
- 同机场优先
- 邻近顺序优先

用于：

- 当前节点删除后的自动迁移
- 分流出口节点删除后的补位

## 名称归一化建议

第一阶段不建议把名称归一化纳入主身份。

原因：

- 不同机场命名风格差异太大
- 归一化规则如果不准，容易把不同节点错误归到同一主身份
- 当前更保守、更稳妥的策略是：
  - 主身份直接用 `source_scope + 原始名称`
  - 改名识别靠“副身份不变”

因此这里的结论改为：

- 名称归一化可以做
- 但应作为后续增强，不应阻塞第一阶段

如果未来要引入，建议：

- 作为可切换版本的 `identity_primary_v2`
- 不直接替换第一阶段规则

## 建议的实现方向

### 第一阶段

先不改现有写库主流程，只增加一份“订阅 identity map”中间结果。

第一阶段主身份规则：

- `primary = hash(source_scope + raw_name)`

不做名称归一化。

这里的 `source_scope` 建议直接定义为：

- 本地节点：`local`
- 订阅节点：`airport_identity + "_" + short_url_hash`

例如：

- `sslinks_78fd`

原因：

- 要支持“同一机场多个账号 / 多个订阅链接”
- 不能只用 group 或域名来做来源识别

输入：

- 订阅前本地节点集
- 订阅后在线节点集

输出：

- `same_node_updates`
- `renamed_nodes`
- `deleted_nodes`
- `new_nodes`
- `replacement_candidates`

### 第二阶段

把这个映射接入：

- `fss_node_current`
- `fss_node_failover_backup`
- 节点分流出口节点引用恢复

### 第三阶段

再把这套身份机制下沉到 Zig 工具层，优先考虑：

- `node-tool`

由它统一负责：

- 节点归一化
- 身份匹配
- 订阅 diff
- 替代节点选择

## 当前建议

短期先做两件事：

1. 订阅日志里明确区分：
   - 仅顺序变化
   - 同名节点参数变化
   - 新增/删除/重命名
2. 为未来 `node-tool` 预留“节点身份匹配”入口，不再把恢复逻辑只绑死在 `name + type + server + port`

## 一句话结论

“节点身份”不是单纯为了日志好看，而是未来以下能力的前置条件：

- 当前节点恢复
- 故障转移节点恢复
- 节点分流出口节点稳定映射
- 订阅 diff 可解释化
