# fancyss 未来待办：SSEP 订阅会话加密落地计划

本文用于记录 `fancyss` 后续落地 `SSEP`（订阅会话加密协议）的实施计划。

当前状态：

- 仅规划
- 尚未实施
- 不作为当前版本承诺

对应标准草案：

- `doc/design/subscription_session_encryption_draft_v1.md`

---

## 一、目标与定位

`fancyss` 落地 SSEP 的目标，不是承诺“节点地址对恶意终端用户绝对不可见”，而是：

1. 避免订阅链接一旦泄露就直接暴露明文节点
2. 让支持 SSEP 的订阅源可以只向支持的客户端返回可解密订阅
3. 降低被浏览器、脚本、普通抓取器直接解析订阅内容的风险
4. 为后续节点地址隐藏、日志脱敏、存储保护提供基础

需要始终接受一个现实：

- 用户如果完全控制自己的终端和客户端进程，仍可能通过改代码、抓包、调试、读内存等方式提取已解密数据

因此，`fancyss` 的实现目标应该是“提高成本、减少明文暴露面”，而不是“承诺绝对不可提取”。

---

## 二、适用范围

本 TODO 默认覆盖：

- `full`
- `lite`
- 传统 URI 行订阅
- 后续可扩展到 `clash` / `sing-box` 类格式

不默认覆盖：

- 立即重写整个订阅架构
- 立即重写整个节点存储架构
- 立即引入不可审计的大型闭源组件

---

## 三、与当前 fancyss 架构的耦合点

落地 SSEP 主要会影响以下区域：

- `fancyss/scripts/ss_node_subscribe.sh`
  订阅下载、识别、解密、缓存、解析入口
- `fancyss/webs/Module_shadowsocks.asp`
  订阅设置、节点展示、日志提示、前端隐藏策略
- `fancyss/scripts/ss_node_common.sh`
  schema2 节点对象字段、备份导出、恢复、兼容映射
- `fancyss/scripts/ss_conf.sh`
  备份与恢复对受保护节点的处理
- `fancyss/scripts/ss_webtest.sh`
  测速、节点环境缓存、输出日志是否泄露真实地址
- `fancyss/ss/ssconfig.sh`
  运行时配置生成时是否会把真实地址暴露到临时文件或日志
- `fancyss/scripts/ss_proc_status.sh`
  状态汇总输出是否泄露受保护节点地址
- `fancyss/scripts/ss_status.sh`
  健康检查和错误日志是否泄露地址
- `fancyss/scripts/ss_node_shunt.sh`
  分流模式下节点信息、测速、缓存、面板展示是否泄露地址

这意味着：

- 这不是一个只改订阅脚本的小功能
- 至少要分阶段推进

---

## 四、总体分期建议

建议分为七个阶段推进。

### 阶段 0：标准定稿与验证前置条件

目标：

- 把协议草案固定到一个最小可实现子集
- 先验证路由器环境能否稳定完成 `32` 字节随机数生成、`HKDF-SHA256`、`AES-256-GCM`

待办：

1. 确认 `fancyss` 第一版到底采用：
   - Header 优先 + UA 兼容
   - 还是先做 UA profile
2. 确认不同平台运行环境中可用的加密实现：
   - `openssl` CLI
   - 自带小型 helper
   - 现有二进制是否可复用库能力
3. 准备跨平台测试向量：
   - 固定 `Session Secret`
   - 固定 `salt`
   - 固定 `iv`
   - 固定明文
   - 固定结果
4. 确认 Envelope JSON 的字段最终命名
5. 明确 `auto` / `protected-only` / `plain-only` 的前端交互方式

验收标准：

- 能在开发环境中用固定向量完成加解密互验
- 能确认最终实现路径不会被某个平台卡死

---

### 阶段 1：订阅下载链路支持 SSEP

目标：

- 在不破坏现有明文订阅逻辑的前提下，为订阅下载增加“发送会话参数、识别 Envelope、完成解密”的能力

核心文件：

- `fancyss/scripts/ss_node_subscribe.sh`

待办：

1. 在订阅下载入口增加模式概念：
   - `plain-only`
   - `auto`
   - `protected-only`
2. 增加会话参数生成函数：
   - 生成 `32` 字节随机 `Session Secret`
   - 生成随机 `req`
3. 改造 `get_ua()` / `download_by_curl()` / `download_by_wget()`：
   - 支持发送 `SSEP` 参数
   - 优先支持独立 Header
   - 如需兼容，再追加 UA profile
4. 增加响应识别：
   - 是否 `Content-Type=application/ssep+json`
   - 是否形如 SSEP Envelope
5. 增加解密步骤：
   - Envelope 解析
   - `req` 校验
   - `HKDF-SHA256`
   - `AES-256-GCM`
   - 可选 `gzip` 解压
6. 解密成功后，把解密结果接回现有订阅解析路径
7. 订阅结束后销毁本次会话参数

注意点：

- 不要把 `Session Secret` 写入 dbus
- 不要把 `Session Secret` 放入全局环境变量
- 不要在日志中打印 Header / UA 中的敏感字段

验收标准：

- 同一个订阅源，在明文模式下行为保持不变
- SSEP 模式下，能成功获取、解密、解析订阅
- 失败时日志能说明是“下载失败 / Envelope 非法 / 解密失败 / 格式不支持”

---

### 阶段 2：订阅缓存与元数据改造

目标：

- 让订阅缓存知道“这是不是受保护订阅”
- 避免现有缓存结构把受保护订阅又意外长期保存成明文

核心文件：

- `fancyss/scripts/ss_node_subscribe.sh`
- `doc/reference/dbus_key_mapping_reference.md`（后续如需补文档）

待办：

1. 为 `parsed/sub_<hash>.meta` 增加字段，例如：
   - `sub_protected=1`
   - `sub_protected_version=1`
   - `sub_protected_mode=auto|required`
   - `sub_protected_alg=A256GCM`
   - `sub_protected_fmt=uri-lines`
   - `sub_protected_last_ok_ts=...`
2. 评估 `raw cache` 是否允许保存解密后的完整原始订阅：
   - 若保存，风险更大但实现简单
   - 若不保存，实现更安全但需要调整缓存恢复逻辑
3. 至少确保：
   - 受保护订阅与普通订阅缓存元数据分离
   - 出错时不会误把密文当明文缓存
4. 订阅日志增加“本次来源是否为受保护订阅”的明确提示

建议：

- 第一版优先不把“已解密的完整原始订阅”长期持久化
- 如果因为现有解析链路必须暂存，应限制在临时目录并在任务结束后清理

---

### 阶段 3：schema2 节点对象增加“受保护来源”标记

目标：

- 节点被成功导入后，前端和后续脚本能知道它来自受保护订阅

核心文件：

- `fancyss/scripts/ss_node_common.sh`
- `fancyss/scripts/ss_node_subscribe.sh`
- `fancyss/webs/Module_shadowsocks.asp`

建议新增的 schema2 元字段：

- `_protected_sub=1`
- `_protected_sub_ver=1`
- `_protected_source=<source_tag>`
- `_protected_fmt=uri-lines|clash-yaml|singbox-json`
- `_protected_mask=server`

说明：

- 第一版不建议一开始就把真实地址加密进 node blob
- 但至少要先把“这个节点来自受保护订阅”记录下来

待办：

1. 订阅写节点时带上上述元字段
2. `fss_node_export_native_backup`
   - 明确是否保留这些字段
3. `restore`
   - 恢复后不能丢失保护标记
4. legacy 兼容层
   - 至少保证不会因为缺少旧字段而导致崩溃

---

### 阶段 4：前端和日志的地址隐藏

目标：

- 用户即使订阅成功，也不要在常规 UI 和日志里默认看到真实服务器地址

核心文件：

- `fancyss/webs/Module_shadowsocks.asp`
- `fancyss/scripts/ss_node_subscribe.sh`
- `fancyss/scripts/ss_proc_status.sh`
- `fancyss/scripts/ss_status.sh`
- `fancyss/scripts/ss_webtest.sh`

建议行为：

1. 受保护节点在节点列表页：
   - 默认不显示真实 `server`
   - 可显示固定占位，如 `protected-node`
   - 或显示短掩码值
2. 节点编辑页：
   - 默认不展示真实地址
   - 对受保护来源节点禁用手工复制服务器地址
3. 订阅日志：
   - 不打印真实服务器地址
   - 只打印节点名、类型、分组、受保护标记
4. 运行状态页：
   - 不回显真实服务器地址
5. 二维码 / 分享功能：
   - 对受保护来源节点默认禁用
   - 或给出明确限制提示

注意点：

- 前端隐藏只是“减小日常暴露面”，不是安全边界
- 后端脚本仍然要继续处理真实地址

---

### 阶段 5：本地存储层保护

目标：

- 进一步减少真实服务器地址在本地 dbus / 缓存 / 导出中的明文暴露

这是高难度阶段，必须谨慎。

原因：

- `fancyss` 运行在路由器上，很多运行期脚本仍要拿到真实地址去生成配置
- 如果本地密钥也存放在本机，那么拥有 root 权限的人仍可提取

因此，这一阶段的定位应是：

- 降低“顺手就能看到明文地址”的概率
- 不宣传为“root 也拿不到”

可评估方案：

1. 只对 `server` / `naive_server` / `hy2_server` 等高敏感字段做字段级保护
2. 使用设备本地随机主密钥：
   - 首次启用时生成
   - 写入受限权限文件
3. `ssconfig.sh` 生成运行配置时短暂解密
4. 备份导出时默认不导出真实地址，或要求显式选择

风险：

- 与当前大量 shell 逻辑耦合严重
- 很容易出现某个角落忘记解密 / 重复解密 / 误打日志

建议：

- 第一版不要和阶段 1 绑定上线
- 等传输层稳定后再单独设计

---

### 阶段 6：运行时、测速、分流、状态面板的泄露审计

目标：

- 确认受保护节点在运行期不会被其它模块重新暴露

需要审计的区域：

- `ss_webtest.sh`
  测速配置、节点环境缓存、日志
- `ss_node_shunt.sh`
  分流规则面板、节点目标列表、运行态缓存
- `ssconfig.sh`
  生成配置时的临时文件和日志
- `ss_proc_status.sh`
  进程状态页
- `ss_status.sh`
  检测脚本输出
- 备份/恢复
- 调试模式和 debug 包

待办：

1. 列出所有会打印 `server` 的地方
2. 列出所有会把 `server` 持久化到文件的地方
3. 给受保护节点统一加一层脱敏输出函数
4. 确认 debug 包不会因为额外日志把真实地址打出来

---

### 阶段 7：生态互通、测试向量和参考实现

目标：

- 让 `fancyss` 实现能成为后续路由器插件和桌面客户端的参考实现之一

待办：

1. 提供最小测试向量
2. 提供 Provider 端示例
3. 提供 Client 端解密示例
4. 给不同客户端整理互通矩阵：
   - `fancyss`
   - `merlinclash`
   - `passwall`
   - `v2rayN`
   - 安卓客户端
   - iOS 客户端
5. 当标准升级到 `v=2` 时，保留 `v=1` 兼容路径

---

## 五、fancyss 第一版建议的最小实现范围

为了避免项目过大，`fancyss` 第一版建议只做以下内容：

1. 订阅下载链路支持 SSEP
2. 受保护订阅识别与解密
3. schema2 节点对象记录“受保护来源”标记
4. 前端节点列表和主要日志隐藏真实地址
5. 普通明文订阅逻辑保持完全兼容

第一版先不强做：

1. 节点对象全量字段级加密
2. 本地 root 场景下的强保护
3. 复杂的多算法协商
4. 大范围重构整个订阅缓存系统

---

## 六、关键实现决策点

正式开工前，至少需要先拍板以下问题。

### 6.1 承载方式

二选一或分阶段：

1. 标准层 Header 优先，`fancyss` 同时做 UA 兼容
2. `fancyss` 第一版先走 UA，后续再补 Header

建议：

- 从标准和长期兼容性看，应是 Header 优先
- 从 `fancyss` 当前改造成本看，可同时保留 UA 兼容

### 6.2 加密实现方式

候选：

1. 直接依赖现有系统 `openssl`
2. 新增一个很小的 helper，只负责：
   - `random`
   - `hkdf`
   - `aead decrypt`

建议：

- 不要先假设每个平台 `openssl` CLI 都足够一致
- 先做验证
- 如果 CLI 兼容性差，宁可引入一个窄职责、小体积、自研但可审计的 helper

这里需要强调：

- “不要单独做复杂加解密程序”是对的
- 但“完全不能有一个很小的协议 helper”不必写死
- 关键是职责要窄、实现要透明、跨平台要一致

### 6.3 受保护节点的展示策略

需要拍板：

1. 完全不显示地址
2. 显示假地址
3. 显示掩码

建议：

- 默认显示掩码或固定占位
- 不建议显示随机假地址

原因：

- 假地址容易引起排障困扰
- 掩码或固定占位更可预期

### 6.4 缓存策略

需要拍板：

1. 解密后的原始订阅是否允许持久化缓存
2. 受保护节点是否允许走现有 parsed cache 恢复逻辑

建议：

- 第一版尽量不持久化解密后的完整原始订阅
- 只缓存必要的解析结果和元数据

---

## 七、建议新增的用户可见配置项

前端可考虑新增：

1. 订阅保护模式
   - 关闭
   - 自动
   - 强制
2. 受保护节点显示策略
   - 隐藏地址
   - 掩码显示
3. 受保护节点导出策略
   - 禁止导出
   - 导出时提醒

注意：

- 第一版不建议暴露过多细节参数
- 不建议让用户手工填写算法、盐、IV 长度之类低层选项

---

## 八、实现顺序建议

推荐实施顺序：

1. 阶段 0：完成标准最小子集确认与可行性验证
2. 阶段 1：订阅下载链路支持 SSEP
3. 阶段 2：缓存与元数据改造
4. 阶段 3：schema2 标记受保护来源
5. 阶段 4：前端与日志地址隐藏
6. 阶段 6：泄露审计
7. 阶段 5：本地存储层保护
8. 阶段 7：生态互通资料整理

把阶段 5 放后面的原因：

- 它复杂
- 风险高
- 对当前运行时耦合最深
- 不应阻塞第一版的传输层落地

---

## 九、最终判断

`fancyss` 落地 SSEP 是值得做的，但应该按“现实边界清楚、分阶段推进”的方式做。

第一版最有价值的成果，不是：

- 承诺任何人都无法从客户端提取真实节点地址

而是：

- 让订阅链接不再天然等于明文节点清单
- 让支持的客户端和服务端先拥有一个统一的受保护订阅传输层
- 为后续 UI、日志、备份、存储保护建立一致语义

如果后续真的开工，建议优先把阶段 0 和阶段 1 做扎实，再决定是否进入更重的存储层保护。
