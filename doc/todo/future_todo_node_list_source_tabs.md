# fancyss 未来待办：节点列表按来源分 Tab

本文用于记录节点列表后续可能推进的一项前端改造：按节点来源拆分列表显示。

当前先记录方案，不立即实施。等“节点分流”主功能落地后，再决定是否继续推进节点列表改造。

---

## 一、目标

当前节点列表把本地节点和多个订阅来源节点全部放在一张表里显示。

当用户订阅机场较多、节点数量较大时，会出现几个问题：

- 单表过长，视觉上臃肿
- 不同机场节点混在一起，查找困难
- 用户在同一机场内查看、编辑、测速时缺少来源维度的聚合视图

本方案的目标是：

- 当节点来源超过 1 个时，按来源拆分为多个 Tab
- 当节点来源只有 1 个时，不显示 Tab，保持当前界面和行为不变
- 尽量只改“视图层”，不改变节点存储结构和现有核心逻辑

---

## 二、范围边界

本方案只覆盖“按来源分 Tab”的第一阶段改造，不包含以下内容：

- 协议筛选
- 名称 / 服务器 / 延迟排序
- 卡片式节点视图
- 节点分流规则编辑页

这些能力如未来仍有必要，再单独设计。

---

## 三、来源定义

节点列表按“来源”拆分时，来源建议定义为：

1. 本地节点
2. 每一个订阅来源

展示名称建议如下：

- 本地节点：固定显示为 `本地节点`
- 订阅来源：显示订阅机场名，优先取当前订阅来源对应的显示组名（即用户实际感知的机场名）

需要注意：

- UI 展示名可以用 `group`
- 但内部来源身份不建议只靠 `group` 判断
- 内部最好继续使用稳定来源标识（如现有订阅来源标识 / source tag / link hash 派生标识），避免不同来源机场名相同导致混淆

也就是说：

- 显示层：用机场名
- 数据层：用稳定来源 ID

---

## 四、首版界面规则

### 1. 单来源场景

如果节点总来源数只有 1 个：

- 不显示来源 Tab
- 节点列表保持当前单表格样式
- 拖拽、编辑、删除、二维码、测速等交互全部维持现状

这包括：

- 只有本地节点
- 只有一个订阅来源
- 没有本地节点、只有一个机场

### 2. 多来源场景

如果节点总来源数大于 1 个：

- 在节点列表上方增加来源 Tab
- 每次只显示当前来源对应的节点表格
- 不同时渲染多张表格堆叠展示

建议的 Tab 顺序：

1. `本地节点`
2. 各订阅来源，按当前来源出现顺序展示

首版是否增加 `全部` Tab，不作为必选项：

- 若追求最小改动，可先不加 `全部`
- 若后续发现用户仍需要总览，再追加 `全部` 视图

当前倾向：

- 首版先不加 `全部`
- 先把“按来源拆开看”这件事做干净

---

## 五、建议的实现原则

### 1. 只改视图层，不改节点实体结构

节点本身仍维持当前 schema 2 / 兼容 schema 1 的存储方式。

Tab 只是前端显示分组，不引入新的节点存储模型。

### 2. 仍然复用当前表格组件

不要做“多套节点表组件”。

建议继续复用当前节点表格生成逻辑，只是在渲染前先根据当前来源筛选出一组节点，再生成表格。

这样可以最大限度复用已有能力：

- 节点编辑
- 节点删除
- 启用节点
- 二维码
- 单节点测速
- 缓存延迟展示

### 3. Tab 只控制“当前视图的节点集合”

切换 Tab 只影响：

- 当前表格渲染哪些节点
- 当前表格显示的节点数量

不应影响：

- 当前主节点
- 节点实际配置
- 订阅来源数据
- 节点测速缓存

---

## 六、需要提前考虑的问题

虽然该方案只是“分 Tab 显示”，但会连带影响一些现有功能，后续实施时需要统一定义。

### 1. 批量测速

这是该方案最容易引发歧义的点。

建议首版规则：

- 批量测速仍保持“全局节点”语义
- 不因为用户当前停留在哪个来源 Tab，就只测速该来源
- 测速结果继续按节点维度缓存

原因：

- 现有延迟测试逻辑本身是全局的
- 如果切 Tab 就改变测速范围，用户容易困惑
- 首版目标是改视图，不改测速语义

对应表现：

- 在 A 来源 Tab 触发批量测速，后台依然测试全部节点
- 当前 Tab 内能实时看到属于该来源节点的结果
- 切到其它 Tab 时，再从缓存或后续结果中展示对应节点延迟

后续如有必要，可再新增：

- 仅测速当前来源
- 测速全部节点

两个入口分开处理。

### 2. 单节点测速

单节点测速不受影响。

当前 Tab 里点哪一个节点，就测哪一个节点。

### 3. 拖拽排序

拖拽排序是多来源 Tab 场景下第二个麻烦点。

原因很直接：

- 当前节点顺序是全局顺序
- 但来源 Tab 只是子集视图
- 在子集里拖拽，无法直观表达“全局顺序如何变化”

因此建议：

- 首版多来源 Tab 场景下，先禁用拖拽排序
- 单来源场景保持当前拖拽能力不变

也就是说：

- 单来源：和现在一样可拖拽
- 多来源：先只做分 Tab 浏览，不开放拖拽

如果未来一定要支持多来源排序，再单独定义规则，例如：

- 仅支持源内排序
- 或单独提供“全局排序视图”

但这不应作为首版目标。

### 4. 节点增删改后的 Tab 刷新

节点编辑、删除、添加、订阅更新后，需要同步处理：

- 重新统计来源
- 更新 Tab 数量和名称
- 如果当前 Tab 来源已被删空，则自动切换到第一个有效 Tab

### 5. 当前主节点高亮

当前主节点可能位于非当前 Tab。

因此建议：

- 当前 Tab 内如果包含主节点，则继续高亮
- 当前 Tab 不包含主节点时，不做额外跳转
- 页面顶部的“当前节点选择”仍照常显示当前主节点

不要因为用户切换 Tab，就强行切回主节点所在来源。

## 七、与未来“节点分流”的关系

后续节点列表大概率不再只是“主节点管理页”，还会承载节点分流相关入口。

因此这个来源 Tab 方案必须兼容未来节点分流。

建议提前保留以下思路：

### 1. 节点列表仍作为“节点资产池”

节点列表负责：

- 展示节点
- 编辑节点
- 启用主节点
- 测速
- 后续为节点分流挑选目标节点

### 2. 节点分流页负责“规则管理”

未来节点分流建议单独在新标签页管理：

- 规则 1：AI 域名 -> 美国节点
- 规则 2：YouTube -> 香港节点
- 默认代理流量 -> 当前主节点

节点列表不直接承载复杂规则编辑，只提供“把该节点用于节点分流”的入口。

### 3. 节点列表后续可能需要增加状态标识

等节点分流功能落地后，节点列表可能要给节点增加轻量状态，例如：

- 当前主节点
- 已被节点分流规则引用
- 默认兜底节点

因此本次来源 Tab 改造应尽量不要把操作列做死，后续仍要留扩展空间。

---

## 八、建议的未来实施顺序

若后续决定推进，建议顺序如下：

### Phase 1：按来源分 Tab

只实现：

- 单来源保持现状
- 多来源显示来源 Tab
- 当前来源表格切换展示
- 批量测速保持全局语义
- 多来源场景先禁用拖拽排序

### Phase 2：与节点分流联动

在节点分流主功能落地后，再决定是否继续扩展：

- 节点列表中的分流状态标识
- 行级“加入节点分流”入口
- 节点被规则引用时的提示信息

### Phase 3：再评估是否继续改节点列表

到这一步再决定要不要加：

- 协议筛选
- 排序
- 搜索
- 全部 Tab
- 其它高级视图

也就是说，来源 Tab 只是第一刀，不应一次性把节点列表做成过重的复杂前端。

---

## 九、当前结论

当前结论如下：

- 节点列表首个值得做的方向，是“按来源分 Tab”
- 单来源场景必须保持当前体验不变
- 多来源场景下，先解决“看起来太乱”的问题，不急着叠加筛选、排序、卡片化
- 批量测速首版保持全局语义
- 多来源场景首版先禁用拖拽排序，避免全局顺序和局部视图冲突
- 等节点分流功能落地后，再决定是否继续对节点列表做第二阶段改造

---

## 十、文档用途

本文仅作为未来待办设计记录，不代表立即进入实现。

后续如果开始做节点列表来源 Tab，实施前应再次核对：

- 当前节点分流设计是否已稳定
- 当前节点操作列是否需要同步调整
- 当前批量测速、单节点测速、拖拽排序的行为是否仍与本文一致
