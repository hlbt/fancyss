# fancyss 未来待办：结合 Xray DNS 的启动链重构

本文用于记录 fancyss 后续在引入更完整的 `xray dns` 能力后，统一重构启动链、DNS 链路和相关运行时逻辑的待办方案。

当前先记录目标、现状、实测基线与未来施工方向，不立即实施。

---

## 一、背景

fancyss 当前启动链在最近几轮已经完成了两类优化：

1. 移除节点服务器预解析，统一改为动态解析
2. 瘦身 `ss_base.sh` / `ssconfig.sh` 的启动装载成本

这两类优化已经明显改善了：

- 点击按钮到第一行日志出现的时间
- 低性能设备上的脚本装载耗时

但从实测看，当前剩余的大头已经主要集中在：

- DNS 启动链
- 当前节点域名运行时解析
- 与 DNS 方案强绑定的后续运行逻辑

也就是说：

- 如果后续要继续显著优化启动时间
- 或进一步统一运行时解析链

就不应再零散修补 `dnsmasq / chinadns-ng / smartdns / 当前节点解析`

而应该在未来引入更完整的 `xray dns` 时，一并重构。

---

## 二、当前实测基线

以下数据来自最近几轮在真实路由器上的实测，作为未来重构前的性能基线。

### 1. 纯脚本装载耗时

#### GS7

- `source ss_base.sh`：约 `1.15s`
- 第一轮瘦身后：约 `1.07s`
- 第二轮拆分前复测：约 `1.25s ~ 1.41s`

- `source ssconfig.sh`：
  - 优化前：约 `1.59s`
  - 优化后：约 `0.16s ~ 0.19s`

#### TUF-AX3000

- `source ss_base.sh`：
  - 优化前：约 `7.89s`
  - 第一轮瘦身后：约 `6.89s`
  - 第二轮拆分后稳定复测：约 `4.41s`

- `source ssconfig.sh`：
  - 优化前：约 `8.81s`
  - 优化后：约 `0.34s ~ 0.35s`

### 2. 真实启动耗时

#### GS7

- 第一轮后：
  - 第一行日志出现：`320ms`
  - 总启动耗时：`17.34s`

- 第二轮后：
  - 第一行日志出现：`210ms`
  - 总启动耗时：`17.57s`

结论：

- GS7 的“首行日志出现时间”已经明显优化
- 总时长瓶颈已主要不在脚本装载，而在后半段运行逻辑

#### TUF-AX3000

- 第一轮后：
  - 第一行日志出现：`2330ms`
  - 总启动耗时：`37.0s`

- 第二轮后：
  - 第一行日志出现：`260ms`
  - 总启动耗时：`30.77s`

结论：

- TUF 上装载优化收益明显
- 但即使如此，后半段启动链仍然很重

### 3. 启动检测耗时

#### GS7

- `check_internet`：`10ms`
- `check_ipv6_proxy_prerequisites`：`10ms`
- `sync_dns_ipv6_policy`：`10ms`
- `check_chn_public_ip`：`440ms`
- `check_frn_public_ip`：`430ms`

#### TUF-AX3000

- `check_internet`：`20ms`
- `check_ipv6_proxy_prerequisites`：`10ms`
- `sync_dns_ipv6_policy`：`20ms`
- `check_chn_public_ip`：`640ms`
- `check_frn_public_ip`：`6970ms`

结论：

- 普通连通性检查并不重
- 真正重的是国内/代理出口 IP 检测
- 而这两项已经受前端开关控制

所以：

- 在不动 DNS 主逻辑的前提下，检测这条线已经没有太多可挖空间

## 三、为什么要等 Xray DNS 一起做

当前 DNS 启动链牵涉太多组件：

- `dnsmasq`
- `chinadns-ng`
- `smartdns`
- 节点域名直连清单
- 机场特调 DNS
- DNS 劫持
- 启动期当前节点域名运行时解析
- `webtest`
- 节点分流 / 运行时统计 / 故障转移

如果在现有结构下继续单点优化，很容易出现：

- 多套解析逻辑并存
- 日志越来越复杂
- 启动顺序越来越难维护
- 每新增一种 DNS 方案，就要多改一套兼容分支

所以更合理的做法是：

- 等未来把 `xray dns` 能力真正用起来
- 再把“域名解析、路由决策、启动顺序、日志、运行时检测”一起重构

---

## 四、未来目标

结合 `xray dns` 的重构，最终希望达到：

1. 启动链更短
2. DNS 组件更少
3. 当前节点运行时解析更稳定
4. 特调机场 DNS 能被统一纳入同一条解析链
5. `webtest / 分流 / 启动 / 故障转移` 对节点域名解析的理解一致

---

## 五、建议的未来架构方向

### 1. 统一“节点域名解析来源”

当前节点服务器域名解析不应再由多个位置各自决定：

- 启动期一套
- `webtest` 一套
- 特调机场一套
- 后续 `xray dns` 又一套

未来目标：

- 所有节点服务器域名解析都走统一抽象层

建议产物：

- 一个清晰的“节点解析计划”
- 明确当前节点应走：
  - `xray dns`
  - `smartdns`
  - `chinadns-ng`
  - 或某种特调上游

### 2. 统一 DNS 运行角色

未来需要明确：

- `dnsmasq` 负责什么
- `xray dns` 负责什么
- `smartdns` 负责什么
- `chinadns-ng` 是否继续保留，保留到什么程度

可能的方向：

- 某些模式下由 `xray dns` 接管更多“代理域名解析”职责
- `smartdns` 更聚焦于直连 / 国内择优 / 特调机场 DNS
- `dnsmasq` 尽量退化为最薄的入口与 DHCP 配套

### 3. 统一“当前节点运行时解析结果”日志

当前用户仍希望保留：

- 启动末尾输出“节点服务器解析地址：x.x.x.x，来源：domain”

未来不应删除这一能力，而应改为：

- 不管底层用的是 `xray dns`、`smartdns` 还是其它方案
- 启动结束时都尽量给出统一格式的最终解析结果

也就是说：

- “观测日志”应该保留
- “预解析写配置”不再保留

### 4. 统一 `webtest` 与运行时 DNS 逻辑

未来 `webtest` 不应自己再维持另一套域名解析理解。

目标：

- `webtest`
- 启动主链
- 节点分流

都应尽量复用同一套最终运行产物或同一套 DNS 计划。

### 5. 机场特调进入统一解析框架

机场特调 DNS 已经存在：

- 特调机场识别
- 特调 DNS 存储
- 特调 smartdns 运行时生成

未来应考虑：

- 不再只把特调视为 smartdns 特例
- 而是把特调机场 DNS 抽象为：
  - 某机场的解析策略覆盖项

使其可以被：

- `xray dns`
- `smartdns`
- 甚至未来其它 DNS 后端

统一消费。

## 六、建议施工顺序

### 阶段 1：只做设计和计时，不改 DNS 主逻辑

需要先补充更细启动计时日志：

- `restart dnsmasq`
- 生成 `chinadns-ng` 配置
- 生成 `smartdns` 配置
- 启动 `chinadns-ng`
- 启动 `smartdns`
- 当前节点运行时解析
- 规则下发

目标：

- 明确未来 DNS 重构真正的大头在哪

### 阶段 2：梳理 DNS 职责边界

输出一份明确设计：

- 哪些域名由哪一层解析
- 哪些结果只用于观测
- 哪些结果进入运行时配置

### 阶段 3：引入 `xray dns` 运行产物

不是马上替换所有现有逻辑，而是先做：

- `xray dns` 配置生成能力
- 与当前节点域名解析 / 特调机场 DNS 的对接

### 阶段 4：选择性替换当前 DNS 链路

按模式逐步替换，而不是一次性全局替换。

例如：

- 先在分流模式验证
- 再在普通模式验证

### 阶段 5：统一 `webtest / 启动 / 分流`

等 `xray dns` 稳定后，再进一步统一：

- `webtest` 域名处理
- 当前节点运行时解析日志
- 机场特调覆盖逻辑

---

## 七、当前不建议做的事

在真正开始 `xray dns` 重构前，不建议继续做这些零散优化：

- 大幅改 `dnsmasq` 重启策略
- 对 `smartdns/chinadns-ng` 做零散的兼容分支叠加
- 再额外引入一套新的节点域名解析缓存语义
- 把特调机场 DNS 做成更多临时例外

原因：

- 这些都会增加未来整合成本

---

## 八、一句话总结

最近几轮优化已经证明：

- 非 DNS 的启动装载成本已经被压掉大头
- 真正剩余的大头主要集中在 DNS 启动链和当前节点运行时解析

因此：

- 如果未来要继续显著优化启动时间
- 最合理的路径不是继续零散修补
- 而是等 `xray dns` 一起，把 DNS、启动顺序、运行时解析和日志统一重构
