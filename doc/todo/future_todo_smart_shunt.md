# fancyss 未来待办：智能分流

本文用于记录 fancyss 后续可能推进的一项功能：智能分流。

这里的“智能分流”不是重新实现一套 Clash/Mihomo/Sing-box 的动态策略组系统，而是借鉴它们在“节点分类”和“规则集合分配出口”上的做法，帮助用户一键生成一套适合当前节点池的 fancyss 节点分流规则。

当前先记录方案与 TODO，不立即实施。

---

## 一、目标

智能分流想解决的问题很直接：

- 用户不需要手动一条条添加节点分流规则
- 用户只需要点击一次按钮，就能自动生成一套可用的节点分流规则
- 规则会基于当前已有节点池做地域分类和节点挑选
- 常见流量集合会被自动分配到更合适的落地地区

它借鉴的主要是 Clash/Mihomo/Sing-box 这类客户端的两个思路：

1. 通过节点名称、分组、关键词、正则，对节点进行地域分类
2. 预先定义一套“规则集合 -> 地域出口”的策略模板

但它不照搬以下能力：

- 不实现 selector
- 不实现长期动态切换策略组
- 不实现“规则集合指向一个可交互组，再由组内继续自动切换”的完整运行时系统

第一阶段只做：

- 一次性生成一套 fancyss 节点分流规则快照

也就是说：

- 智能分流 = 自动帮用户产出当前可直接生效的 `ss_basic_shunt_rules`
- 而不是 = 在路由器里重做一套 Clash 策略组引擎

---

## 二、范围边界

第一阶段的智能分流只覆盖：

- 前端入口
- 策略选择弹窗
- 节点地域分类
- 节点挑选
- 自动生成 fancyss 节点分流规则
- 预览与应用

第一阶段不做：

- selector / url-test / fallback 这种长期动态组
- 持续后台自动换节点
- 用户自定义正则分类编辑器
- 第三方配置导入器
- 广告过滤全量策略
- 复杂的个人化策略市场

原因很明确：

- fancyss 当前已经有成熟的“节点分流规则”系统
- 第一阶段最重要的是尽快把“一键生成规则”做出来
- 不应为了追求与 Clash 一致，提前把复杂度拉到不可控

## 三、产品形态

### 1. 前端入口

建议在节点分流页的“添加节点分流”卡片旁边，增加一个新的“智能分流”卡片：

- 同样使用虚线框
- 与加号卡片同尺寸、同层级
- 图标建议使用机器人 / 魔棒 / 自动化符号

也就是：

- `+` 卡片：手动添加规则
- `机器人` 卡片：智能生成规则

### 2. 点击后的交互

点击智能分流卡片后，弹出配置弹窗。

弹窗首版建议包含：

- 策略选择
- 策略说明
- 节点挑选方式
- 兜底方式说明
- 预览结果
- 应用按钮

### 3. 弹窗主要字段

首版建议字段如下：

1. 智能策略
2. 节点挑选方式
   - 随机
   - 延迟最低
3. 兜底方式
   - 直连
   - 代理
   - 由策略决定
4. 预览区域
   - 识别到的地域节点
   - 每条规则最终使用哪个节点
   - 哪些地域发生了回退

说明：

- “节点挑选方式”先只支持随机和延迟最低即可
- “手动为每个地域指定节点”可以放在后续迭代

## 四、核心设计原则

### 1. 借鉴规则匹配思路，不照搬 selector

智能分流要借鉴 Clash/Mihomo/Sing-box 的是：

- 节点分类方式
- 规则集合与地域出口的绑定方式
- 策略模板的产品表达
- “GFW catch-all” 这种补漏思路

但不直接照搬：

- selector
- url-test 组
- fallback 组
- group 嵌套 group
- 长期运行时自动重选组内节点

第一阶段更适合 fancyss 的表达方式是：

- 把智能策略“展开”为普通的 fancyss 节点分流规则

这非常重要，因为它决定了：

- 现有节点分流、热重载、webtest、规则维护系统都可以继续复用

### 2. 仍然输出当前 fancyss 已支持的规则模型

第一阶段不要新建第二套运行时结构。

最终输出仍然是：

- `ss_basic_shunt_rules`
- `ss_basic_shunt_default_node`

智能分流只是“自动生成器”，不是“第二套引擎”。

### 3. 要明确引入 GFW catch-all

如果某个策略的最终兜底是 `DIRECT`，则必须在 `DIRECT` 前增加一条：

- `GFW / proxy / 被墙域名集合 -> 代理节点`

否则很多未被显式命中的被墙网站会直接掉到 `DIRECT`，造成不可访问。

因此：

- `GFW catch-all` 是智能分流里一个必须显式存在的补漏规则
- 它应该放在最终兜底规则之前

### 4. 手动节点不能一刀切判死刑

首版不建议简单限制“只有订阅节点才能用智能分流”。

更合理的判断是：

- 只要节点名称 / 分组 / 标识中能识别出地域，就可以参与智能分流

也就是说：

- 订阅节点天然更适合做智能分类
- 手动节点如果命名规范，也完全可以参与

真正应该限制的是：

- 当可识别地域节点过少
- 或某个策略的关键地域缺失过多

这时提示用户当前节点池不适合该智能策略，而不是只按“是否订阅”粗暴限制。

## 五、节点分类模型

### 1. 可用数据来源

智能分流节点分类可优先使用这些字段：

- `name`
- `group`
- （必要时）`server`

其中推荐优先级：

1. `name`
2. `group`
3. `server`

原因：

- `name` 最接近用户感知
- `group` 对某些机场有机场级聚类价值
- `server` 只作为兜底，不适合当主要分类依据

### 2. 第一阶段建议支持的地域桶

首版建议支持这些地域类别：

- `hk`
- `sg`
- `tw`
- `jp`
- `kr`
- `us`
- `uk`
- `eu`
- `other`

其中真正第一批可用重点建议先围绕：

- `hk`
- `sg`
- `tw`
- `us`
- `jp`

因为这些是实际最常见、最容易在策略里使用的出口。

### 3. 分类方式

分类建议采用“正则 + 关键词白名单”的方式。

例如：

- `hk`
  - `香港|HK|Hong ?Kong|HKG|🇭🇰`
- `sg`
  - `新加坡|SG|Singapore|SIN|🇸🇬`
- `tw`
  - `台湾|TW|Taiwan|TPE|🇹🇼`
- `us`
  - `美国|US|USA|America|Los Angeles|New York|🇺🇸`
- `jp`
  - `日本|JP|Japan|Tokyo|Osaka|🇯🇵`

为了减少误判，建议做一个“归一化”过程：

- 全部转小写
- 去除空格、下划线、多余分隔符
- 常见 emoji / 国家缩写单独识别

### 4. 分类输出

分类结果不直接暴露为 selector，而是先生成一个内部结果：

```json
{
  "hk": ["122", "123", "132"],
  "sg": ["163", "166"],
  "tw": ["153", "154"],
  "us": ["173", "174"],
  "jp": ["144", "145"]
}
```

后续策略挑选只基于这个分类结果。

## 六、节点挑选方式

### 1. 首版支持两种

第一阶段建议只支持：

- 随机
- 延迟最低

### 2. 随机

适用于：

- 节点延迟缓存还不完整
- 用户只想快速生成一套规则

实现上很简单：

- 在目标地域桶中随机选一个节点

### 3. 延迟最低

适用于：

- 节点已有 webtest / latency 缓存
- 用户更希望选择当前可用且速度更好的节点

首版可直接复用现有延迟结果：

- 优先使用已有缓存
- 若无缓存，再降级为随机

不要在第一阶段为了智能分流强制触发一次全量测速。

理由：

- 用户点击“智能分流”后等待时间会显著变长
- 逻辑会变复杂
- 与“快速一键生成”的初衷不符

### 4. 缺失地域时的回退

策略里必须允许设置地域回退链。

例如：

- `us -> sg -> hk -> current`
- `sg -> hk -> us -> current`
- `hk -> sg -> tw -> current`

这样即使用户没有某个国家节点，也还能生成一套可用策略。

## 七、策略模型

策略本身建议独立成一个配置文件，例如：

- `smart_profiles.json`

每个策略包含：

- `id`
- `label`
- `description`
- `default_mode`
- `selection_mode_default`
- `rules`
- `fallback_chain`

示意：

```json
{
  "id": "balanced_direct",
  "label": "均衡分流（直连兜底）",
  "description": "常见流量按地域出口分配，最后直连兜底，并补一条 GFW 规则保证被墙域名可访问。",
  "default_mode": "direct",
  "rules": [
    {"preset": "ai", "region": "us"},
    {"preset": "google", "region": "hk"},
    {"preset": "telegram", "region": "hk", "fallback": ["sg"]},
    {"preset": "youtube", "region": "sg", "fallback": ["us"]},
    {"preset": "media", "region": "sg", "fallback": ["hk"]},
    {"preset": "gfw", "region": "hk", "fallback": ["sg", "us"]}
  ]
}
```

## 八、第一批策略建议

这里专门按你刚才指出的问题修正。

### 1. 均衡分流（直连兜底）

适合：

- 日常使用
- 尽可能把常见流量分到更合适的出口
- 其它未命中流量仍然直连

建议规则：

1. `ai -> us`
2. `google -> hk`
3. `telegram -> hk / sg`
4. `youtube -> sg / us`
5. `media -> sg / hk`
6. `gfw -> hk / sg / us`
7. `default -> direct`

注意：

- 这里的 `gfw` 必须在 `default -> direct` 之前

### 2. 代理兜底

适合：

- 用户更在意“不要漏代理”
- 未分类流量也希望默认走代理

建议规则：

1. `ai -> us`
2. `google -> hk`
3. `telegram -> hk / sg`
4. `youtube -> sg / us`
5. `media -> sg / hk`
6. `default -> current_proxy`

说明：

- 该策略下 `gfw` 可选
- 因为兜底已经是代理，`gfw catch-all` 的补漏价值会下降
- 但如果想显式为被墙域名指定更优地域，也可以保留 `gfw -> hk / sg`

### 3. 保守直连

它不能和“均衡分流”做成一回事。

它应该更保守，规则更少，只处理最典型的需要代理集合：

1. `ai -> us`
2. `telegram -> hk / sg`
3. `youtube -> sg / us`
4. `media -> sg`
5. `gfw -> hk / sg / us`
6. `default -> direct`

和“均衡分流”的差异在于：

- 不主动给 `google`
- 不主动给更多平台类集合
- 规则数量更少
- 更强调“少改动、少接管”

### 4. 流媒体优先

适合：

- 用户更关注 YouTube / 流媒体体验

建议规则：

1. `youtube -> us / sg`
2. `media -> sg / us`
3. `ai -> us`
4. `telegram -> hk / sg`
5. `gfw -> hk / sg / us`
6. `default -> direct`

### 5. 广告过滤策略

第一阶段不建议上线。

原因：

- 你当前还没有完整的广告类规则集体系
- 后续你还计划做 AB 两套规则集
- 广告过滤策略更适合等规则集体系稳定后再接入

也就是说：

- `全量媒体`
- `全量 + 广告过滤`
- `某某个人优化策略`

这些都更适合第二阶段再做。

---

## 九、规则顺序原则

智能分流生成规则时，顺序必须明确。

建议首版统一采用：

1. 屏蔽类规则（如果该策略包含）
2. 明确指定出口的高优先级集合
   - `ai`
   - `telegram`
   - `youtube`
   - `media`
   - `google`
3. `gfw catch-all`
4. `default`

说明：

- 如果最终兜底是 `DIRECT`，则 `gfw catch-all` 必须存在
- 如果最终兜底是 `PROXY`，则 `gfw catch-all` 可选

## 十、与 Clash/Mihomo/Sing-box 的关系

智能分流可以明确借鉴这些客户端的以下思路：

### 1. 借鉴的部分

- 节点地域分类
- 规则集合分配不同地区出口
- GFW / Media / AI / Google / Telegram 等集合化处理
- “漏网之鱼”兜底的策略表达
- “全量直连兜底 / 全量代理兜底”两种主流风格

### 2. 不照搬的部分

- selector
- url-test 组
- fallback 组
- group 嵌套 group
- YAML/JSON 配置解释器

也就是说：

- 借鉴的是“域名/IP 匹配规则的做法”
- 不是把整个 Clash 策略组模型搬进来

## 十一、后端实现建议

建议新增一个独立脚本，例如：

- `fancyss/scripts/ss_smart_shunt.sh`

建议动作：

1. `preview`
2. `apply`

### 1. `preview`

负责：

- 校验当前节点池是否满足策略要求
- 做地域分类
- 做节点挑选
- 生成将要应用的规则草案
- 返回给前端预览

### 2. `apply`

负责：

- 基于预览结果写入：
  - `ss_basic_shunt_rules`
  - `ss_basic_shunt_default_node`
  - `ss_basic_shunt_rule_ts`

它本质上只是一个“自动生成规则”脚本。

### 3. 推荐复用的现有能力

可以直接复用：

- 节点数据读取：
  - `ss_node_common.sh`
- 延迟/可用性缓存：
  - `ss_webtest.sh`
- 节点分流规则结构：
  - `ss_node_shunt.sh`

## 十三、可用性校验

智能分流不是所有场景都能直接用。

首版建议校验这些条件：

1. 当前主模式是 `xray 分流模式`
2. 至少存在若干可用于分流的节点
3. 至少识别到策略所需的关键地域节点之一

不建议做成：

- “只要不是订阅节点就完全禁用”

更合理的是：

- 分类结果为空时禁用
- 关键地域严重不足时提示不建议使用

## 十四、阶段划分

### 阶段 0：规划

- 明确策略模型
- 明确分类规则
- 明确节点挑选方式

### 阶段 1：MVP

- 后端 `preview/apply`
- 前端机器人卡片
- 2 到 4 个官方内置策略
- 随机 / 延迟最低 两种选点方式

### 阶段 2：增强

- 允许用户手动指定某个地域使用哪个节点
- 允许保存用户最后一次使用的策略
- 增加更多集合与更多地域

### 阶段 3：高级策略

- 接入广告过滤策略
- 结合 AB 规则集
- 增加更细的回退链配置

## 十七、当前建议结论

基于 fancyss 当前的架构，智能分流完全可以做，而且应该做成：

- 借鉴 Clash/Mihomo/Sing-box 的“节点分类 + 规则集合分配出口”的做法
- 不照搬 selector
- 首版只做“一键生成规则快照”

这是当前最稳、最贴合 fancyss、也最容易落地的方案。
