# future todo: status-tool evolution

## 背景

`fancyss` 早期的状态检测链路是：

1. 前端进入页面
2. 前端主动通知后端开始检测
3. 后端执行轻量 shell + curl 检测
4. 检测结果立即回传前端展示

这种模式的优点很明确：

- 用户不在 `fancyss` 页面时，不做无意义的状态检测
- 结果即测即显，前端展示内容和本次检测结果严格一致
- 后端资源占用与用户行为强相关，符合“按需检测”的初始设计

后续为了支持故障转移，引入了后台周期性检测：

- 用户不在页面时，后台也要能持续得到状态结果
- 一旦发现异常或延迟超过阈值，可以切换节点 / 重启插件 / 关闭插件

这导致状态系统从“纯前端触发”演进成了“前端检测 + 后台检测”双源系统。

## 当前已暴露的问题

### 1. shell + curl 路径过重

为了提高状态准确性，`ss_status.sh` 后来逐渐叠加了：

- 国内 / 国外并发检测
- IPv4 / IPv6 分支
- warmup 预热
- score1 / score2 重试
- 远端地址 / 错误码 / 调试信息

结果是：

- 进程数大幅增加
- shell 启动和 `source ss_base.sh` 成本变高
- 弱机型如 `TUF-AX3000` 很容易在轮询间隔内跑不完
- 前端刷新、后台轮询和故障转移检测互相叠加时，会形成明显 CPU 峰值甚至进程积累

### 2. 前后端两套状态源的时间颗粒不一致

后台是随机化或区间化的周期检测，前端又有自己的刷新节奏。

这会导致：

- 前端展示的并不一定是“刚刚测出来”的结果
- 前端如果坚持主动轮询后端接口，就会制造额外压力
- 如果靠 `httpd/_api` 高频轮询，华硕环境下 `httpd` 稳定性可能受影响

### 3. `ws` 很有价值，但不能假设永远可用

现有 `ws` 并不只是日志通道，而是一个类似“轻量命令执行 API”的信道：

- 前端可通过 `ws` 发命令
- 后端脚本真正执行命令
- 执行结果通过 `ws` 回传

这使得：

- 前端状态检测完全可以走 `ws`
- 只要同一条信道长期保持，就不需要依赖 `httpd/_api` 高频轮询

但是必须明确：

- `ws` 不是所有情况下都适用
- 浏览器、网络、中间代理、页面生命周期等都会让 `ws` 失效
- 所以后端仍然必须保留 `httpd api / _temp 文件` 的保障机制

结论：

- `ws` 应视为高效通道
- `httpd api` / `_temp` 文件应视为兜底通道
- 两者必须长期并存，而不是单选

## status-tool 的长期目标

`status-tool` 的目标不能只停留在“替换 `ss_status.sh`”。

它应该成为：

- 状态检测统一引擎
- 后台状态缓存提供者
- 前端主动检测执行器
- 故障转移状态来源
- 未来 `webtest` 的底层网络探测基础组件

也就是说，`status-tool` 的设计从一开始就要考虑：

- `status` 模式
- `webtest` 模式
- 后台 daemon 模式
- 前端 on-demand 模式
- 与 `ws` / `httpd api` 的交互模式

## 分阶段演进方案

### 阶段 A：过渡期

目标：

- 用 `status-tool once` 替换最重的 shell + curl 状态探测
- 用 `status-tool daemon` 提供后台缓存
- 前端优先读缓存，不再高频制造真实探测

当前阶段已经基本完成的方向：

- `status-tool once`
- `status-tool daemon`
- `ss_status.sh` 精简成轻量包装
- `ss_status_main.sh` 优先消费后台缓存

但这不是最终形态，只是止血与收敛阶段。

### 阶段 B：恢复按需检测语义

目标：

- `failover = 0` 时，不做无意义后台周期检测
- 前端进入页面时才触发状态检测
- 用户离开页面后停止前端状态检测

建议策略：

- 未开启故障转移：
  - 优先使用 `status-tool once`
  - 前端按自己的节奏发起检测
  - 检测结果通过 `ws` 回传，失败时走 `httpd api` / `_temp` 文件兜底

- 开启故障转移：
  - 使用 `status-tool daemon`
  - 后台持续检测并维护缓存
  - 前端只读取缓存，不再主动触发真实检测

这是一个“once / daemon”双模式并存的稳定版本，可以先落地。

### 阶段 C：引入 serve 模式

如果验证后发现：

- `status-tool once` 在弱机型上虽然比 shell 轻很多，但频繁启动/退出仍有可感知峰值

那么就进入下一阶段：`serve` 模式。

`serve` 模式目标：

- `status-tool` 常驻，但默认空闲
- 只有收到命令时才开始探测
- 可切换成持续探测 / 停止探测 / 单次探测
- 一个进程内部完成状态管理，尽量避免反复启动新进程

## serve 模式通信方案约束

这部分是后续设计重点，必须提前明确原则。

### 原则 1：不能只依赖 ws

原因：

- `ws` 是浏览器侧通道，不是系统级 IPC
- 后台故障转移、shell 脚本、计划任务不适合依赖浏览器侧 `ws`
- 所以 `serve` 模式和浏览器之间可以通过 `ws` 间接交互，但 `serve` 自身与系统的通信方式必须独立存在

### 原则 2：必须有 httpd api / 文件兜底

即使以后前端主要通过 `ws` 使用 `status-tool serve`，也必须保留：

- `httpd api`
- `_temp` 状态文件

原因：

- 路由器页面刷新、重连、浏览器兼容性等场景下都需要兜底
- 不能让“没有 ws”就导致页面完全失能

### 原则 3：IPC 必须足够轻

通信方式候选：

1. Unix domain socket
   - 优点：轻量、低延迟、适合请求-响应、可承载结构化协议
   - 缺点：需要实现最小协议，但这其实是可接受成本

2. FIFO / named pipe
   - 优点：实现简单
   - 缺点：双向通信、并发和状态管理较弱，不适合复杂请求

3. signal (`kill -SIGXXX`)
   - 优点：极轻
   - 缺点：只能表达极少量语义，不适合携带探测参数或返回结果

4. 辅助小二进制作为 client
   - 优点：可把协议/参数封装干净
   - 缺点：如果本质上还是走 socket，那么 client 是否独立存在要看复用价值

当前建议：

- `serve` 模式优先使用 Unix domain socket
- 不建议把 `signal` 作为主通信方式
- `signal` 最多只适合做“停止/重载”这类极少量控制语义
- 如果后续发现 shell 端频繁拼 socket 协议仍然麻烦，再考虑加一个更小的 `statusctl` 客户端二进制

结论：

- 首选：`status-tool serve` + Unix socket
- 备选：必要时增加极薄的 `statusctl`
- 不建议：把 `signal` 作为主协议

## 建议的 serve 命令语义

后续可考虑：

- `probe_once`
- `get_cache`
- `start_watch`
- `stop_watch`
- `get_state`
- `reload_config`

其中：

- 前端短驻留页面可以用 `probe_once`
- 故障转移可用 `start_watch`
- 页面关闭或不需要持续检测时可用 `stop_watch`
- 前端刷新页面时优先 `get_cache`

## 与 webtest 的关系

这是 `status-tool` 设计里必须提前考虑的重点。

未来有较大概率会将 `webtest` 的 curl 请求也逐步移除，转而复用这一工具的探测能力。

潜在方向有两种：

1. 同一个 `status-tool` 扩展多 profile
   - `status`
   - `webtest`
   - 甚至未来的其它网络探测

2. 同一个二进制双开进程
   - 一个实例负责状态检测
   - 一个实例负责 webtest

当前建议：

- 优先做成同一二进制多 profile
- 由命令或协议区分：
  - `status-tool once --profile status`
  - `status-tool once --profile webtest`
  - `status-tool daemon --profile status`

原因：

- 复用同一套 socket / http / socks5 / dns / timer / 错误码实现
- 降低后续维护成本
- 避免每做一个网络子功能就再造一套工具

需要提前保证的设计点：

- 错误码和结果结构必须可扩展
- 状态探测结果与 webtest 结果要能共存，但不能混淆
- 缓存文件 / socket 命名要按 profile 分离

## 当前施工建议

短期不要直接上 `serve`，建议顺序是：

1. 先把 `failover=0 -> once`、`failover=1 -> daemon` 路线彻底稳定
2. 在 `TUF-AX3000` 这类弱机型上实测：
   - `once` 高频触发时的 CPU 峰值
   - 页面停留 1-3 分钟时的资源变化
3. 如果 `once` 反复拉起仍有明显峰值，再进入 `serve` 设计和施工

## 结论

`status-tool` 的最终方向不是简单替换一个 shell 脚本，而是成为：

- 状态检测统一引擎
- 故障转移状态来源
- 前端按需检测执行器
- 未来 `webtest` 的底层网络探测基础

但演进必须分阶段进行：

- 先稳定 `once/daemon`
- 再视性能收益决定是否引入 `serve`
- 无论走到哪一步，`ws` 都不是唯一通道，必须保留 `httpd api` / `_temp` 文件兜底
