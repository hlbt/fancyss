# 关于推进订阅加密的倡议

副标题：面向路由器插件、桌面客户端、移动客户端与订阅服务提供方的讨论稿

当前状态：

- 倡议文档
- 讨论稿
- 不是标准
- 不是最终协议

相关参考：

- `doc/design/subscription_session_encryption_draft_v1.md`
- `doc/todo/future_todo_subscription_session_encryption.md`

## 一、我们为什么要提出这个倡议

近两年，中转型线路越来越常见。

这类线路的基本形态通常是：

- 用户先连接中国大陆境内的中转入口
- 中转入口再通过更稳定的出境链路与境外落地服务器通信

这类方案的优势很明显：

- 延迟和丢包体验通常更好
- 稳定性往往优于直接连接境外服务器
- 对普通用户而言，接入门槛也更低

但这类方案也有一个越来越现实的问题：

- 订阅内容一旦以明文形式下发，中转入口的域名、地址和相关基础设施就很容易被直接解析出来

而目前主流订阅，大多仍然是以下几类形式：

- Base64 文本
- 明文 URI 列表
- 明文 YAML / JSON

这意味着：

- 只要拿到订阅链接
- 再按常规方式请求和解析
- 就能较低成本地还原出节点地址、域名和部分中转基础设施信息

一旦这种暴露形成规模化，后果通常不是“单个节点偶发失效”，而是：

- 中转入口被持续识别
- 线路资源被频繁更换
- 上游资源承压
- 服务方和用户都被迫陷入高频替换与修复

问题发展到今天，已经不只是“机场怎么做得更稳”的问题，而是：

- 订阅内容本身是否还应该继续以可直接解析的明文方式下发

## 二、为什么这个问题不能只靠机场单方面解决

很多人会自然想到一个方向：

- 那就让服务端把订阅内容加密后再下发

但真正落地时，很快会遇到一个现实限制：

- 如果客户端不主动参与，服务端无法单方面完成这件事

原因很简单：

1. 客户端最终仍然要拿到可解析、可导入、可运行的节点配置。
2. 如果服务端返回的是密文，客户端必须具备解密能力。
3. 如果解密所需的密钥是固定的、公开的、长期存在的，那么它很快就会退化为“换了一层包装的明文”。
4. 因此，更合理的方式只能是：
   - 由客户端在每次订阅时生成一次性会话密钥
   - 服务端基于这次会话密钥加密订阅
   - 客户端只在本次订阅过程中短暂解密并使用

也就是说：

- 订阅加密这件事，必须由客户端先发起
- 机场可以选择跟进
- 但客户端如果完全不支持，机场也无法独立把这件事做成

对于使用开源客户端的用户来说，这一点尤其关键：

- 如果客户端生态没有共识
- 服务端就算愿意尝试，也很难形成稳定兼容的实现

所以我们发起这份倡议，不是为了要求某一家机场先改，而是希望：

- 各类客户端先把“受保护订阅”的入口能力做出来
- 让服务端真正拥有可跟进的基础

---

## 三、这份倡议想推动的，不是“终极保密”，而是“先减少明文暴露”

需要先把预期讲清楚。

我们推动订阅加密，并不是在宣称：

- 只要做了加密，终端所有者就永远无法拿到节点地址

这在现实中并不成立。

因为只要用户完全控制自己的设备、客户端进程和运行环境，那么理论上仍然可能通过：

- 抓取自己发出的请求
- 修改客户端调试输出
- 读取运行内存
- 查看解密后的运行配置

来提取最终已经被客户端使用过的数据。

所以，订阅加密真正能解决的是另一类更现实、更普遍的问题：

- 订阅链接泄露后，第三方无法再直接低成本获取明文节点清单
- 浏览器、脚本、通用下载器、普通解析器，不能再把订阅当作现成明文内容处理
- 服务端可以只对“明确支持该机制的客户端会话”返回可解密订阅

换句话说，第一步不是追求“绝对不可见”，而是：

- 先把“拿到订阅链接就等于拿到明文基础设施信息”的现状改掉

这一步本身就有现实意义。

## 四、我们建议的基本思路

为了便于不同客户端尽快落地，我们建议先推动一个足够简单、足够实用的试点思路。

核心流程如下：

1. 用户把订阅链接粘贴到客户端中。
2. 客户端发起订阅请求时，临时生成一段一次性随机会话密钥。
3. 客户端把“我支持订阅加密”以及“本次会话参数”一并发送给服务端（利用UA，或者请求header承载）。
4. 服务端检测到客户端支持该机制后，不再返回传统明文订阅，而是返回一份加密封装订阅。
5. 客户端收到后，仅在本次订阅流程中解密。
6. 解密后的内容继续走客户端现有的订阅解析逻辑。
7. 订阅完成后，本次会话密钥销毁，不持久化保存。

这一思路的关键点有三个：

- 密钥是客户端主动提供的
- 密钥是一次性的
- 解密只发生在本次订阅会话里

## 五、我们建议优先讨论的“最小草案”

这份倡议不是标准，但为了推动实际讨论，我们仍然建议先围绕一个最小可落地的草案展开。

在当前阶段，我们更倾向于把它理解为：

- 一个可以先试点的 `v0` 方案
- 而不是一开始就写得很重、很完整的正式标准

### 5.1 基本原则

`v0` 试点版建议遵守以下原则：

1. 不破坏现有明文订阅
2. 允许机场逐步跟进，而不是一次性强制切换
3. 允许不同客户端先做最小互通，再逐步增强
4. 尽量不引入复杂的用户配置项
5. 优先保护订阅传输层，再考虑 UI、日志、存储层

### 5.2 会话参数由客户端主动提供

客户端每次刷新订阅时：

- 生成一段一次性随机值，作为本次订阅会话密钥
- 生成本次请求的随机 `req id`

这两个值只应在当前进程内存中短暂存在，不应：

- 写入配置
- 写入数据库
- 写入环境变量
- 写入日志

### 5.3 承载方式建议

如果从长期演进看，更理想的方式是：

- 首选：独立 HTTP Header
- 兼容：`User-Agent` 附加 profile

但如果考虑当前很多路由器插件和机场面板的现实情况，`v0` 试点阶段我们建议：

- 优先从 `User-Agent` 方案开始
- 后续再逐步演进到独立 Header

原因很现实：

- 很多路由器端订阅本身就是 `curl + wget`
- 客户端实现上，加 `UA` 几乎没有门槛
- 很多机场面板现在已经能够识别 `UA`
- 相比之下，很多面板短期内并没有现成的“按自定义 Header 识别订阅行为”的能力

这里也需要明确：

- 从长期来看，把会话参数直接放进 `UA` 并不是理想方案
- 因为 `UA` 很容易被 CDN、反代、日志系统完整记录

所以我们的倡议是：

- `v0` 可以从 `UA` 先行
- 后续版本再逐步把承载方式过渡到独立 Header

### 5.4 `v0` 试点版的最小做法

为了让更多客户端和机场面板先跑起来，我们建议 `v0` 尽量简单：

1. 客户端在订阅请求时，在 `UA` 中追加一段约定字段
   - 例如：`subkey=xxxxxxxxxxxxxxxx`
2. `subkey` 为本次订阅的一次性随机值
3. 服务端检测到 `UA` 中存在 `subkey` 后，按约定算法加密订阅内容
4. 客户端收到后，使用本次 `subkey` 解密
5. 订阅完成后，`subkey` 即销毁

第一阶段的关键目标不是把协议设计得多完整，而是：

- 先让“客户端主动提供一次性会话密钥”这件事真正跑起来

### 5.5 服务端返回加密封装订阅

服务端检测到客户端支持后，返回的不再是传统明文订阅，而是：

- 带版本号
- 带算法标识
- 带格式标识
- 带必要的随机参数
- 带密文与认证标签

的一个轻量封装对象。

客户端解密后，再继续按原有方式解析：

- URI 行
- Base64 行
- Clash YAML
- sing-box JSON

### 5.6 默认算法建议

如果只讨论 `v0` 的实用方案，我们建议优先考虑：

- 对称加密
- `AES-256-GCM`
- 外层使用可传输的文本封装格式

为了让讨论尽量聚焦，`v0` 可以先遵守一个很简单的约定：

- 只要没有显式指定其它方式，就默认使用 `AES-256-GCM`

原因不是因为它“唯一正确”，而是因为：

- 成熟
- 跨平台实现广
- 适合作为最小试点子集

### 5.7 关于实现方式

很多路由器端订阅下载，本质上就是：

- `curl`
- `wget`

所以从落地角度出发，我们完全理解很多项目会优先希望：

- 用现有工具链就能实现
- 不增加新的复杂二进制依赖

因此，`v0` 阶段我们建议：

- 可以优先提供 `OpenSSL`、`PHP`、`Python`、`Go` 等常见实现示例
- 让面板和客户端作者可以直接套用

但这里仍然有一个需要注意的现实：

- 文档可以提供 `openssl` 命令示例
- 但协议本身最好不要完全写死为某个特定命令行工具的默认行为

更稳妥的方式是：

- 倡议文档里给出常见语言和工具的参考实现
- 草案层只约定输入输出和默认算法

### 5.8 兼容模式建议

客户端侧建议至少提供三种模式：

- `plain-only`
  只接受传统订阅
- `auto`
  优先尝试受保护订阅，失败再回退
- `protected-only`
  要求服务端必须返回加密订阅

这能兼顾：

- 老订阅继续使用
- 新订阅渐进部署
- 需要更严格策略的用户也有明确开关

## 六、为什么我们呼吁“客户端先支持”

这份倡议最想强调的一点是：

- 订阅加密不是机场一方可以单独完成的改造

如果客户端没有实现“发送会话参数”和“解密返回内容”的能力，那么机场就算愿意尝试，也只能面对以下问题：

- 大量客户端直接失效
- 同一条订阅需要维持多套分发逻辑
- 没有统一字段
- 没有统一兼容约定
- 没有统一错误处理方式

这会让服务端迟迟不敢动。

所以我们真正希望的是：

- 客户端先把能力入口做出来
- 服务端再逐步启用
- 社区再围绕互通细节形成第一版实用草案

只有这样，订阅加密才有机会从概念变成生态能力。

## 八、这份倡议希望达成的共识

我们希望推动的，不是某个项目的私有扩展，而是一个尽量简单、尽量中立、尽量可复用的共识。

理想状态下，第一阶段只需要形成以下共识就足够有价值：

1. 客户端应支持一种“受保护订阅”模式。
2. 客户端在订阅请求时应主动生成一次性会话密钥。
3. `v0` 可以优先通过 `UA` 承载会话参数。
4. 服务端可根据客户端声明，选择返回传统订阅或加密订阅。
5. 第一版只解决“订阅传输层”的保护，不强求一次解决所有问题。
6. 各客户端之间尽量共用字段语义，而不是各写一套互不兼容的私有协议。

只要这五点能先达成，后面很多增强能力都可以再逐步补：

- 前端隐藏真实地址
- 日志脱敏
- 导出限制
- 存储层保护
- 设备绑定
- 更严格的会话策略

## 九、我们建议的推进方式

为了避免讨论停留在概念层，我们建议按以下顺序推进：

### 第一步：先把倡议讲清楚

让更多客户端作者理解：

- 问题到底在哪里
- 为什么机场单方面做不了
- 为什么客户端必须先加入

### 第二步：先形成一个最小草案

先不要追求“大而全”，而是先约定：

- `UA` 里会话参数怎么传
- 响应内容怎么封装
- 默认算法先用什么
- 失败如何回退

### 第三步：至少有两到三个客户端做 PoC

只要有不同平台的客户端能先互通，整个事情就会从“概念”进入“可执行”阶段。

### 第四步：邀请机场和面板方跟进

当客户端已经具备入口能力后，服务端再决定：

- 是否提供受保护订阅
- 是否按来源逐步开启
- 是否同时保留传统订阅

这样迁移阻力会小很多。

## 十、这份倡议不是为了制造封闭，而是为了减少不必要的明文暴露

我们尤其想强调一点：

- 订阅加密不是为了把客户端做成黑盒
- 也不是为了排斥开源
- 更不是为了人为提高正常用户的使用门槛

相反，它真正想解决的是：

- 当前订阅内容对中转基础设施的暴露过于直接
- 这种暴露方式已经不适合今天的实际环境

如果客户端生态继续把明文订阅视为默认前提，那么服务端就很难主动改善这一点。

因此，这份倡议本质上是在呼吁：

- 用一个跨平台、可讨论、可审计、可渐进部署的方式
- 先把“订阅内容明文暴露”这个基础问题往前推一步

## 十一、结语：希望从讨论，走向第一版实用草案

这份文档不是标准，也不试图替所有项目做决定。

它只是想发出一个尽量明确的呼吁：

- 订阅加密这件事，值得被各客户端认真讨论
- 客户端如果不先行动，机场很难单方面解决问题
- 只要多个项目愿意先做最小互通，第一版实用草案就有机会真正出现

如果你是以下任一角色，我们都真诚欢迎你加入讨论：

- 路由器插件作者
- 桌面客户端作者
- iOS / Android 客户端作者
- 面板作者
- 订阅服务提供方
- 协议实现者
- 长期维护订阅解析、节点导入、配置分发相关代码的开发者

我们并不期待第一版一步到位。

但我们期待：

- 从今天开始，至少让“客户端主动发起受保护订阅”成为一个被认真推进的方向
